================================================================================
ATHLEAT — V3 MEAL CAROUSEL: WHAT GOES INTO THE LLM PROMPT (AND A SAMPLE LAYOUT)
================================================================================
Generated for developers. Source files:
  • System + brain: backend/src/services/kez/composer.js + masterPrompt.js
  • User message + catalog: backend/src/controllers/kezController.js (mealCarouselPost)
  • LLM call: backend/src/services/kez/llm.js

--------------------------------------------------------------------------------
1) FOOD DETAILS SHARED IN THE PROMPT
--------------------------------------------------------------------------------

A) VERIFIED MEALS SUMMARY (text lines — NOT full meal objects)
   Built in kezController.js from the top-scored DB meals (legacy meals stack).
   One line per suggested verified card, format:
     "- {title}: P {protein}g C {carbs}g F {fat}g {energy_kj}kJ"
   Example:
     "- Chicken rice bowl: P 35g C 72g F 12g 2850kJ"
     "- Greek yoghurt parfait: P 18g C 45g F 8g 1650kJ"
   The LLM does NOT receive the full ingredient list for these lines — only
   title + rounded macro totals + kJ.

B) ITEMS CATALOG (bulk of tokens when Kez must "gap fill" AI meals)
   Built by buildItemsCatalog() in kezController.js from public.items:
     • FOOD CATEGORIES — comma-separated id|name for up to 200 rows from
       public.food_categories
     • FOOD SUB-CATEGORIES — comma-separated id|title for up to 200 rows from
       public.sub_categories
     • ITEMS table — up to hardLimit rows (default 1000 in code), each line:
         id | title | category | protein | carbs | fat | kJ | serving
       (pipe-separated; kJ parsed from items.energy; serving from serving_size
       + serving_size_unit)
   Only rows where is_locked = false. Ordered by category, title. If the
   joined text exceeds maxChars (default 60_000), lines are dropped from the
   tail until under budget (minimum ~200 lines kept in the shrink loop).

C) ATHLETE PREFERENCES (not per-item rows, but text lists inside FACTS JSON)
   • liked_foods — merged DB preferences + coach-typed list from the request
   • dislikes — derived from prescreen (dislike_foods etc.)
   These steer which catalog lines the model should favour or avoid; they are
   not a second copy of the items table.

--------------------------------------------------------------------------------
2) MEAL / CONTEXT DETAILS SHARED IN THE PROMPT
--------------------------------------------------------------------------------

A) SYSTEM MESSAGE (Anthropic "system" or OpenAI system role)
   = MASTER_SYSTEM_PROMPT (masterPrompt.js: Kerry identity, voice rules,
      macros, Henry EER, meal photo protocol, V3 carousel rules, hard stops)
   + buildBrainInjection(...) from composer.js:
        • "=== RUNTIME HARD STOPS & CORRECTIONS ===" from knowledge_entries
          (type 'never' and 'correction')
        • "=== KNOWLEDGE SNIPPETS (retrieved) ===" from pgvector
          match_knowledge_chunks (default 6 chunks, full chunk content)

B) USER MESSAGE — starts with v3CarouselUserPrompt() which includes:

   (i) V3_CAROUSEL_TASK_SUFFIX — JSON schema + high-level carousel rules

   (ii) "ATHLETE & CONTEXT:" + factsJson
        JSON built in mealCarouselPost (analysisFacts), sample shape below.

   (iii) "VERIFIED MEALS CANDIDATES ..." + verifiedSummary (see section 1A)

   (iv) If AI gap-fill is needed (need > 0), appended blocks:
        • "WHEN GENERATING NEW MEALS:" + bullet instructions (read context,
          match V1/V2, preferences, dietary reqs, combine 3–6 catalog items,
          image_prompt, foods with item_id, etc.)
        • itemsCatalog (section 1B)
        • "Generate exactly {need} meals as JSON..."

   factsJson typically includes:
     firstName, mission_id, slot_id, slot_label, slot_guidance, based_on,
     meal_category, need_count,
     athlete { first_name, age, sex, weight_kg, height_cm, goals,
               biggest_challenges, meal_priority, weight_trend, eating_style,
               dietary_requirements, medical_flags, cooking_skills },
     preferences { liked_foods, fav_foods_raw, disliked_foods,
                   disliked_foods_raw },
     training { load_day, activity_type, days_low/med/high, session_length },
     daily_eer { load_day, kcal_low/high, protein/carb/fat gram ranges },
     target_band { kcal_low/high, kj_low/high, p/c/f _low/_high for THIS slot },
     analysis_macros { from meal_analysis.macro_totals for based_on version },
     dislikes, liked_foods (aliases),
     current_meal_v1 { description (trimmed meal_text), image_url },
     improved_meal_v2 { description, image_url }

NOTE ON TOKEN SIZE
------------------
The Anthropic error "208955 tokens > 200000" usually means the SYSTEM side
(MASTER_SYSTEM_PROMPT + very large knowledge chunk hits) plus the USER side
(items catalog + FACTS) exceeded the model limit. Mitigations (if not already
in your branch): cap knowledge chunk length/total, shrink the items catalog
(slot filter, fewer columns), or route this endpoint to OpenAI long-context.
